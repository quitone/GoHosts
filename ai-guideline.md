好的，这是一份 **AI 代码生成准则**，采用思维链（Chain of Thought）结构编写，旨在指导 AI 在生成代码时遵循高内聚低耦合、函数式风格、测试驱动开发、小步快跑迭代、代码风格统一及清晰注释的原则。

```markdown
# GoHosts AI 代码生成准则

## 0. 核心原则总览（执行前必读）

在生成任何代码片段之前，请严格遵循以下优先级顺序：

1. **测试先行**：先写测试，再写实现。
2. **函数式优先**：使用纯函数，避免共享状态和副作用。
3. **高内聚低耦合**：每个模块/函数只做一件事，通过接口通信。
4. **小步快跑**：每次只生成一个可独立验证的功能单元。
5. **风格一致**：严格遵循项目既定的代码风格配置。
6. **注释即文档**：为导出的函数、类型、复杂逻辑提供清晰注释。

---

## 1. 测试驱动开发（TDD）执行流程

### 1.1 生成顺序（强制）

每次功能请求，按以下顺序生成代码：

1. **第一步：生成测试用例**
   - 仅生成测试代码，不包含实现。
   - 测试应描述期望的行为，而非实现细节。
   - 测试必须包含失败断言（明确期望值）。

2. **第二步：生成最小实现**
   - 仅生成让测试通过的**最少代码**。
   - 禁止添加测试未覆盖的任何额外逻辑。

3. **第三步：重构建议**
   - 在测试通过后，提供可选的代码重构建议（不直接生成重构后的实现，除非用户明确要求）。

### 1.2 测试用例编写规范

- **Go 测试**：使用 `testing` + `testify/assert`，测试文件命名 `*_test.go`。
- **前端测试**：使用 `Vitest` + `Vue Test Utils`，测试文件命名 `*.spec.ts`。
- **测试结构**：遵循 `Arrange-Act-Assert` 模式。

**示例（Go）：**

```go
func TestHostsService_Read(t *testing.T) {
    // Arrange: 准备测试环境
    tmpDir := t.TempDir()
    testPath := filepath.Join(tmpDir, "hosts")
    content := []byte("127.0.0.1 localhost")
    os.WriteFile(testPath, content, 0644)
    service := NewService(testPath)

    // Act: 执行被测方法
    got, err := service.Read()

    // Assert: 断言结果
    assert.NoError(t, err)
    assert.Equal(t, string(content), got)
}
```

## 2. 高内聚低耦合设计约束

### 2.1 模块划分原则

- **单一职责**：每个 Go 包、Vue 组件只负责一个明确的功能领域。
- **接口隔离**：Go 包之间通过小型接口通信，而非直接依赖具体实现。
- **依赖倒置**：通过构造函数注入依赖，便于测试替换。

### 2.2 禁止事项

- **禁止循环依赖**：`backend` 下的包之间不能相互引用。
- **禁止全局变量**：业务逻辑中禁止使用 `var` 定义的包级全局变量（配置常量除外）。
- **禁止跨层调用**：前端组件不能直接操作后端服务，必须通过 `services/` 层封装。

### 2.3 推荐结构

```
backend/
├── hosts/       # 仅负责 Hosts 文件读写
├── config/      # 仅负责配置 JSON 的存取
├── fetcher/     # 仅负责远程内容拉取
└── scheduler/   # 仅负责任务调度（依赖 fetcher 和 hosts 的接口）
```

---

## 3. 函数式编程风格要求

### 3.1 纯函数优先

- **无副作用**：函数不应修改输入参数，不应依赖外部可变状态。
- **显式返回值**：通过返回值传递结果，而非通过指针修改参数。

**示例（Go）：**

```go
// ✅ 推荐：纯函数
func MergeHosts(base, overlay string) string {
    // 合并逻辑，不修改原字符串
    return base + "\n" + overlay
}

// ❌ 禁止：有副作用
func MergeHosts(base *string, overlay string) {
    *base = *base + "\n" + overlay // 修改了输入
}
```

### 3.2 避免共享状态

- 使用 `context.Context` 传递请求范围的值，而非全局变量。
- 在 Vue 中，使用 `Pinia` 的 `store` 统一管理状态，组件内部避免直接修改 `props`。

### 3.3 函数签名简洁

- 函数参数不超过 3 个，超过时使用结构体封装。
- 函数体长度不超过 50 行（测试函数除外），超过则拆分。

## 4. 小步快跑迭代约束

### 4.1 单次生成范围

- **一次只生成一个函数/方法**及其对应的测试。
- 如果功能复杂，请主动拆分为多个子步骤，引导用户逐步确认。

### 4.2 生成内容格式

每次生成时，清晰标注：

```markdown
## 当前步骤：实现 `HostsService.Read` 方法

### 1. 测试代码（失败阶段）
[测试代码块]

### 2. 实现代码（最小实现）
[实现代码块]

### 3. 验证命令
```bash
go test ./backend/hosts -v
```
```

## 5. 代码风格一致性规范

### 5.1 强制遵循项目配置

- **Go**：必须通过 `goimports` 格式化，通过 `golangci-lint` 检查。
- **前端**：必须通过 `Prettier` 格式化，通过 `ESLint` 检查。
- 生成代码前，请回忆项目根目录的 `.editorconfig`、`.golangci.yml`、`.prettierrc.json` 配置。

### 5.2 命名约定

| 语言 | 元素 | 风格 | 示例 |
| :--- | :--- | :--- | :--- |
| Go | 包名 | 小写单数 | `hosts`, `config` |
| Go | 导出的函数/类型 | 大驼峰 | `NewService`, `SchemeType` |
| Go | 非导出的函数/变量 | 小驼峰 | `parseLine`, `backupDir` |
| TypeScript | 接口/类型 | 大驼峰 | `Scheme`, `FetchOptions` |
| TypeScript | 变量/函数 | 小驼峰 | `loadHosts`, `schemeList` |
| Vue | 组件文件 | 大驼峰 | `HostsView.vue`, `SchemeManager.vue` |

### 5.3 导入顺序

- **Go**：标准库 → 第三方库 → 项目内部包，分组空行分隔。
- **前端**：第三方库 → 项目内部模块，按字母序排列。

## 6. 注释规范

### 6.1 必须添加注释的场景

| 场景 | 注释要求 | 示例 |
| :--- | :--- | :--- |
| 导出的函数/类型（Go） | 必须以函数名开头，描述行为 | `// NewService creates a new Hosts service with the given path.` |
| 复杂业务逻辑 | 解释“为什么”，而非“做什么” | `// 使用原始写文件而非 txeh，以保留用户自定义注释格式。` |
| 公共接口方法（前端） | JSDoc 格式描述参数和返回值 | `@param schemeId - 要应用的方案 ID` |

### 6.2 禁止的注释

- **无意义注释**：`// 循环遍历列表`（代码已自解释）
- **过期注释**：与当前代码行为不符的注释
- **被注释掉的代码**：应直接删除，而非注释保留

## 7. 生成代码前的自检清单

在生成任何代码之前，请在心里过一遍以下问题：

- [ ] 我是否先提供了测试用例？
- [ ] 这个函数是否只做了一件事？
- [ ] 这个函数是否可以写成纯函数？
- [ ] 是否有副作用或依赖了全局状态？
- [ ] 是否符合项目的命名和格式规范？
- [ ] 是否为导出的符号添加了注释？

如果任一答案为“否”，请调整生成策略。
